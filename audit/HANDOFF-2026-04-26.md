## 9) Module Registry enterprise improvements (Sprint 3+, NO Sprint 1)

Mejoras al patrón Pattern 4 (Module Registry interno) sin splittear plugins. **~30h total**, todas Sprint 3+:

| # | Mejora | Esfuerzo | Beneficio |
|---|---|---|---|
| 1 | Module Manifest (`module.php` head con hooks/deps/config/caps/REST/cron) | 4-6h | Auditabilidad + linter conflictos hook priority |
| 2 | Module Bootstrap Lifecycle (activate/deactivate/uninstall/migrate) | 6-8h | Clean state, no orphan cron/options |
| 3 | Module Dependency Graph (`->dependsOn(EmailModule::class)`) | 3-4h | Fail loud si dep missing |
| 4 | Module Health Endpoint (admin page mostrando status por módulo) | 4h | Observability sin Sentry overhead |
| 5 | Module Test Harness (PHPUnit por módulo en `tests/Modules/<name>/`) | 8-10h | Coverage real por módulo |
| 6 | Module Observability Tags (Sentry tag `module=...`) | 2h | Filtrar Sentry errors por módulo |

**Disparador natural:** cuando agregues el próximo módulo nuevo (ocasión natural para definir el manifest pattern y aplicarlo retroactivamente).

**NO hacer en Sprint 1-2:** Sprint 1 = cleanup + security + setup foundations (beneficio inmediato). Sprint 3+ = polish enterprise (beneficio diferido).
